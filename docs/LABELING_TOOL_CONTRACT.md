# Labeling-tool data contract — ML repo → `satellite-aoi-data-aggregator`

_From the ML/training repo (`truck-detection`). Every fact below is verified against the real
`Annotations-RGB.gpkg` and the exact read logic in `src/export_coco.py` — not paraphrased._

**Verdict:** Green light on the in-app pixel-space labeler. The `L.CRS.Simple` design (label in
`(col, row)`, apply the GeoTIFF affine only at save) is the right call and dissolves the cost driver —
this is days of work, not weeks. Keep emitting a **scene-scoped point layer**, not COCO; the ML side
owns COCO conversion + chipping. The only real risk is data fidelity, and it reduces to two things:
**(1) geometric accuracy** (keypoints land on the right pixels) and **(2) scene linkage** (each vehicle
joins to the correct GeoTIFF). Everything else is mechanical field-matching.

---

## Emit exactly this

A **GeoPackage** named so the exporter finds it, containing one layer:

| Property | Required value |
|---|---|
| Layer / table name | **`Annotations`** (exact) |
| Geometry column | **`geometry`**, type **`POINT`** (not MultiPoint) |
| CRS | **EPSG:32610** (WGS 84 / UTM 10N), coords in **metres** |
| Rows | **one row per keypoint** (a vehicle = 3 rows) |

Fields (exact names, types, encodings):

| Field | Type | Rule |
|---|---|---|
| `fid` | INTEGER PK | Auto-generated by GPKG. **Ignored** by the pipeline — nothing to do. |
| `vehicle_id` | int | Groups the 3 points of one vehicle. **Unique within a scene**; may repeat across scenes. |
| `sequence` | int | **Exactly `1`, `2`, `3`.** Capture order → **`1 = blue (band 2)`, `2 = red (band 6)`, `3 = green (band 4)`**. Place point 1 on the blue blob, 2 on red, 3 on green. |
| `scene` | text | Join key to the GeoTIFF. **See "The #1 risk" below — this is the one that silently breaks.** |

No other field is read. No class, confidence, notes, or timestamp is consumed (single class is hardcoded
in the exporter). Extra fields are harmless but ignored.

**Prefer GPKG over GeoJSON — not for taste.** GeoJSON (RFC 7946) assumes WGS84/4326, so 32610 metre
coordinates in a `.geojson` get silently misread as lat/lng by many tools. GPKG stores `srs_id = 32610`
explicitly. If you need GeoJSON, tell us — it's a one-line exporter change on our side, but you must then
carry the CRS out-of-band.

## The reassuring core (verified true)

The model **does not train on the annotation file.** Training pixels are read **only** from the SR
GeoTIFF at export time:

```python
# src/export_coco.py
with rasterio.open(tif) as src:
    r = src.read(RED); g = src.read(GREEN); b = src.read(BLUE)   # bands 6,4,2 from the .tif
    inv = ~src.transform                                          # world→pixel, from the .tif
```

The annotation supplies **only** location + order + scene. A homemade tool **cannot alter the training
imagery** — it can only (a) place a keypoint on the wrong pixel, or (b) join a vehicle to the wrong tile.
That's the whole risk surface.

## The #1 risk: the `scene` join string

The exporter resolves `scene` → `imagery/all_geotiffs/{scene}.tif`. Our filenames are the **short renamed
stem**, not Planet's:

```
scene = "Stanwood_08_20260504"          ✅   <Location>_<NN>_<YYYYMMDD>
NOT     "20260509_194436_34_250a"       ❌   Planet scene id
NOT     "..._3B_AnalyticMS_SR_8b_clip"  ❌   full Planet stem
```

**Your app stores Planet-named files; our exporter looks up short stems.** If the labeler writes the
Planet name, the exporter prints `! no GeoTIFF for scene ... -- skipping` and drops **every** vehicle —
a silent, total loss with no error.

**Resolution — pick one:**
- (a) the labeler writes the short stem (needs the app to hold the rename map), **or**
- (b) we agree on a canonical key (e.g. the Planet scene id) and the ML side adds an alias/rename so the
  exporter resolves it. **We prefer (b)** — send us the filenames your app actually has and we'll adapt
  the join. The geometry/affine is identical either way; the risk is purely the string.

## Save-time math (world coordinates + the Y-flip guard)

At save, per keypoint: `world = affine · (col, row)` using the tif's transform (rasterio exposes it as
`src.transform` = `(a, b, c, d, e, f)`):

```
x = c + a·col + b·row        # for our north-up scenes: a = +3.0, b = 0  →  x = c + 3·col
y = f + d·col + e·row        #                          e = −3.0, d = 0  →  y = f − 3·row
```

The **`e` term is negative** (pixel height). That minus sign is the single most important line of code in
the tool.

### Three fidelity foot-guns to build against

1. **Row / Y-flip.** Raster pixel space is **row 0 = top, row increases downward**. `L.CRS.Simple` is a
   math plane with **y increasing upward**. If you read a marker's y as "from the bottom," every keypoint
   comes out **vertically mirrored**. Ensure `row` = pixels-**from-top** of the native grid.
2. **Native-resolution render.** The labeling canvas image must be **1 canvas pixel = 1 GeoTIFF pixel** —
   no resampling or fit-scaling. If the served PNG is scaled to bounds, `(col,row)` no longer maps through
   the affine. (`image-rendering: pixelated` at full res is correct; just don't let it get fit-scaled.)
3. **Keep `(col,row)` as floats.** QGIS stores continuous coords; the exporter uses sub-pixel `~transform`.
   Integer pixel-snapping reads as a systematic offset in the parity diff.

## Invariants the exporter enforces (so the tool can match them)

- **Exactly 3 points** per vehicle, `sequence ∈ {1,2,3}`. `0/1/2` fails the `sorted(seqs) == [1,2,3]` check.
- **Incomplete vehicles are silently dropped** (counted, not errored). Partial saves are safe — they just
  won't train. Your call whether to allow saving partials.
- **Order-independent.** The exporter sorts by the `sequence` field, so row order in the file is irrelevant.
- **Whitespace is stripped** on `scene`, and **blank `scene` rows are dropped** (both were real bugs — a
  stray leading space once split a phantom scene).
- The **bbox is derived from the 3 keypoints** and the **chip is centered on their centroid**, so keypoint
  placement accuracy = chip-centering accuracy. Chip size is an export-side knob (`--chip`, default 64);
  road masking is **not** applied by the exporter today — do **not** bake either into the labels.

## Acceptance handshake (prove parity, don't assume it)

Before the tool labels anything for real:

1. **Round-trip diff.** Label one already-labeled scene in the new tool; the ML side diffs the two
   GeoPackages — geometry within **sub-pixel**, identical `vehicle_id / sequence / scene`, CRS = 32610.
   (This is where the Y-flip and float-precision issues surface.)
2. **Real exporter run.** The ML side runs `export_coco.py` **unchanged** on the tool's output and confirms
   valid COCO keypoints + a spot-checked chip **centered on the vehicle**.

## What the ML side will provide

- A **`validate_annotations.py`** the save endpoint can call: checks Point geometry, `sequence ∈ {1,2,3}`,
  complete triples, CRS = 32610, and that every `scene` resolves to a real tif — so the tool **fails loudly
  at save**, not silently at export.
- A **relaxed exporter**: configurable layer name + optional GeoJSON, and the **`scene`-alias resolution**
  once you send your filenames.
- Fixtures + the round-trip diff harness for test #1.
